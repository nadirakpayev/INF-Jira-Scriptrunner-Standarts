# 5.4 Чек-лист обновления Jira и ScriptRunner

> Эта страница фиксирует, что именно нужно проверить до и после обновления Jira или ScriptRunner, чтобы не сломать рабочие скрипты.

## Для чего нужна эта страница

Апгрейды чаще всего бьют по таким местам:

- устаревшие API;
- listeners, jobs и REST endpoints;
- общие модули;
- интеграции с внешними системами;
- mail handlers;
- Assets / AQL сценарии;
- логирование и операционная наблюдаемость.

Эта страница нужна, чтобы апгрейд не превращался в ручной поиск проблем уже в production.

## Что считаем high-risk контуром

В первую очередь проверяем:

- scheduled jobs;
- listeners;
- REST endpoints;
- mail handlers;
- общие модули;
- клиентские интеграции;
- `Assets / AQL` сценарии;
- миграционные и операционные скрипты, которые могут массово менять данные.

## Фаза 1. До апгрейда

### 1. Соберите список критичных скриптов

До начала обновления должен быть список:

- критичных интеграций;
- критичных jobs/listeners;
- переиспользуемых модулей;
- клиентских сценариев с высоким бизнес-риском.

### 2. Зафиксируйте текущую картину

Перед апгрейдом полезно зафиксировать:

- какие скрипты активны;
- где у них код и страницы;
- какие из них уже имеют предупреждения или известные ограничения;
- какая сейчас нормальная длительность и failure rate у критичных jobs.

### 3. Проверьте Script Registry на staging или текущем контуре

До апгрейда нужно увидеть:

- type-checking errors;
- предупреждения об устаревших API;
- общий список критичных артефактов.

Это дает базовую точку сравнения.

### 4. Подготовьте экспорт конфигурации

Если на инстансе активно используются extension points ScriptRunner, перед апгрейдом полезно сделать экспорт конфигурации через `Configuration Exporter`.

Это особенно полезно для:

- listeners;
- REST endpoints;
- других элементов, где важна конфигурация через UI.

### 5. Проверьте временные и спорные решения

Перед апгрейдом отдельно посмотрите:

- временные скрипты, которые все еще активны;
- клиентские обходные решения;
- решения без явного владельца;
- скрипты без нормальной документации.

Это типичные кандидаты на проблемы после обновления.

## Фаза 2. После апгрейда на staging

### 1. Снова откройте Script Registry

После обновления обязательно проверьте:

- новые type-checking errors;
- новые предупреждения об устаревших API;
- скрипты, которые перестали выглядеть корректно;
- общие модули, от которых зависят другие сценарии.

### 2. Проверьте критичные execution paths

На staging нужно прогнать минимум:

- один listener-сценарий;
- один scheduled job;
- одну внешнюю интеграцию;
- один Assets/AQL-сценарий;
- один REST endpoint, если они используются;
- один customer-specific скрипт из high-risk контура.

### 3. Проверьте execution history и логи

Если сценарий формально выполняется, это еще не значит, что он работает нормально.

Нужно проверить:

- нет ли нового роста времени выполнения;
- нет ли аномального количества warnings;
- нет ли новых частичных ошибок;
- не изменился ли формат expected log path.

### 4. Проверьте конфигурационные изменения

Если после апгрейда что-то переопределялось или переподключалось, проверьте audit trail и конфигурационный контур:

- listeners;
- jobs;
- endpoint configuration;
- notes и связи с документацией.

### 5. Проверьте custom REST endpoints

Для custom endpoints отдельно проверьте:

- что endpoint вообще доступен;
- что конфигурация не потерялась;
- что package path и script root scanning работают ожидаемо;
- что доступ к endpoint по-прежнему ограничен нужными группами.

## Фаза 3. Перед выходом в production

Перед выходом в production должно быть ясно:

- какие high-risk сценарии уже проверены;
- какие известные проблемы остаются;
- есть ли временные ограничения после релиза;
- кто будет первым ответственным, если что-то пойдет не так.

Если это не определено, процесс управления обновлением слабый.

## Фаза 4. После выхода в production

### Первые часы после обновления

Смотрите в первую очередь:

- рабочие логи;
- execution history;
- признаки роста времени выполнения;
- повторяющиеся ошибки интеграций;
- клиентские сценарии с внешними зависимостями.

### В первый день после обновления

Нужно проверить:

- нет ли тихих частичных сбоев;
- не изменилось ли поведение mail handlers;
- не сломались ли Assets/AQL запросы;
- не появились ли ошибки только на боевых данных;
- не вырос ли объем WARN/ERROR по критичным скриптам.

## Когда обновление считается проблемным

Обновление считается проблемным, если:

- появились новые type-checking errors;
- появились новые предупреждения об устаревших API в критичном контуре;
- критичный listener/job формально запускается, но ведет себя иначе;
- увеличилось время выполнения или failure rate;
- интеграция теряет идемпотентность;
- никто не может быстро показать, как проверить проблемный сценарий.

## Что фиксировать на страницах скриптов

Если после апгрейда был проверен или изменен скрипт, на странице стоит зафиксировать:

- дату проверки;
- кем проверено;
- какой сценарий проверяли;
- результат;
- ограничения или follow-up;
- был ли нужен отдельный фикс после обновления.

## Минимум без бюрократии

Минимально после каждого существенного апгрейда должны быть сделаны эти вещи:

1. Проверен `Script Registry`.
2. Проверены критичные jobs/listeners/integrations.
3. Проверены логи и execution history.
4. Зафиксированы известные проблемы.
5. Назначен первый ответственный на период после обновления.

## Чего не делаем

- Не идем в production без проверки high-risk сценариев на staging.
- Не игнорируем deprecation warnings, если скрипт критичен.
- Не считаем "скрипт не упал" достаточным критерием успешной проверки.
- Не выпускаем обновление без плана, кто будет смотреть первые ошибки.
- Не проводим обновление, если критичный контур неизвестен.

## Короткий чек-лист после обновления

- Script Registry проверен.
- Critical jobs/listeners проверены.
- Интеграции проверены.
- Assets/AQL сценарии проверены.
- Execution history и логи просмотрены.
- Известные проблемы зафиксированы.
- Ответственный за первые инциденты назначен.
