# 4.5 Уровни логирования

> Логи должны помогать понять, что произошло, а не просто создавать шум.

## Обязательные уровни

| Уровень | Когда использовать |
|---|---|
| `DEBUG` | Для детальной технической диагностики, не как основной рабочий лог |
| `INFO` | Для старта, завершения и ключевых этапов нормального сценария |
| `WARN` | Для частичных проблем, которые не ломают весь сценарий |
| `ERROR` | Для ошибок, из-за которых сценарий не выполнился или выполнился некорректно |

## Примеры по уровням

| Ситуация | Уровень | Пример сообщения |
|---|---|---|
| Запустилась синхронизация | `INFO` | `GLPI sync started: source=glpi, target=assets` |
| Обработано 120 объектов, 3 пропущено | `INFO` | `GLPI sync finished: processed=120, skipped=3, errors=0` |
| Письмо пришло без ключа задачи | `WARN` | `Mail handler skipped message: Jira key not found in subject` |
| Один объект не обновился, остальные обработаны | `WARN` | `GLPI sync partial failure: externalId=GLPI-4421, reason=asset not found` |
| Сценарий упал и не завершился | `ERROR` | `GLPI sync failed: step=updateAsset, externalId=GLPI-4421` |
| Нужна техническая детализация для диагностики | `DEBUG` | `Resolved payload fields: externalId=GLPI-4421, schema=Device` |

## Что обязательно логировать

- старт выполнения;
- завершение выполнения;
- ключевые входные параметры, если это безопасно;
- количество обработанных объектов, если это batch-сценарий;
- ключевые причины ошибок;
- итоговый статус выполнения.

## Что рекомендуется логировать

- шаг сценария, на котором возникла проблема;
- идентификатор сущности;
- количество обработанных объектов для batch-сценария;
- причину пропуска, если сценарий сознательно ничего не сделал.

## Что нельзя логировать без необходимости

- чувствительные данные;
- огромные объекты целиком;
- повторяющийся шум в цикле без пользы;
- внутренние детали, которые не помогают диагностике.

## Чего не делаем

- Не пишем `INFO` на каждый элемент в большом цикле.
- Не используем `ERROR` для штатных пропусков и ожидаемых исключений.
- Не оставляем сообщения вида `something went wrong`.
- Не логируем пароль, токен, полный payload или весь объект целиком без необходимости.

## Формат полезного сообщения

Сообщение должно отвечать минимум на два вопроса:

- что происходило;
- на каком шаге произошла проблема.

Хорошо, если видно:

- тип операции;
- сущность;
- идентификатор;
- итог.

## Минимальный шаблон полезного лога

Используйте такую простую формулу:

`[что делали] + [с чем работали] + [на каком шаге] + [итог]`

Например:

- `AMX device sync started: client=Azercell`
- `Mail handler skipped message: reason=no issue key`
- `Asset update failed: externalId=GLPI-4421, step=mapAndSave`

## Operational note

Для production-сценариев полезно помнить, что диагностика живет не только в логах:

- Script Registry помогает видеть type-checking errors и deprecated API;
- Execution History помогает видеть длительность и сбои выполнения;
- audit trail ScriptRunner помогает понимать конфигурационные изменения.

Логи должны быть частью общей operational-модели, а не единственным источником правды.
