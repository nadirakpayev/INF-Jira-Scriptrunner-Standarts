# 3.2 Структура хранения страниц в Confluence

> Эта страница задает, как размещать страницы по скриптам и связанным материалам в Confluence.

## Главный принцип

Структура в Confluence должна строиться не только по клиентам, но и по смыслу использования:

- переиспользуемые материалы отдельно;
- клиентские материалы отдельно;
- временные и проектные решения не должны выглядеть как вечные стандарты.

## Рекомендуемая логика

```text
ScriptRunner
  Общие решения
  Клиентские решения
  Операционные скрипты
  Миграции
  Архив и устаревшие материалы
```

## Пример page tree

```text
ScriptRunner
  Общие решения
    Интеграции
      GLPI -> Jira Assets
    Общие модули и подходы
      Mapping helpers
  Клиентские решения
    Azercell
      AMX -> Jira
  Операционные скрипты
    Mail handlers
      Mail Handler for B2B
  Миграции
    Jira 11
  Временные и разовые решения
    2026-04
  Устаревшие материалы и архив
```

## Что должно лежать на странице скрипта

- назначение;
- владелец;
- контекст;
- тип скрипта;
- путь к коду;
- зависимости;
- точки запуска;
- шаги проверки;
- риски;
- история изменений.

## Когда создавать отдельную страницу

Отдельная страница обязательна, если:

- скрипт рабочий и используется регулярно;
- скрипт влияет на данные или автоматизацию;
- скрипт интеграционный;
- скрипт может быть изменен другим человеком позже.

Временный одноразовый скрипт тоже должен иметь страницу, если его запуск несет риск для данных или требует повторного запуска в будущем.

## Какую ветку выбирать

- `Общие решения` — для переиспользуемых подходов, модулей и сценариев, которые полезны вне одного клиента.
- `Клиентские решения` — для логики конкретного клиента.
- `Операционные скрипты` — для служебных сценариев команды, поддержки и эксплуатационных работ.
- `Миграции` — для отдельного migration-контекста с ограниченным жизненным циклом.
- `Временные и разовые решения` — для временных сценариев, которые не должны выглядеть как вечные стандарты.
- `Устаревшие материалы и архив` — для неактуальных материалов.

## Чего не делаем

- Не кладем клиентские страницы в общий раздел только потому, что они "могут пригодиться".
- Не держим временные страницы рядом с активными evergreen-материалами.
- Не делаем несколько параллельных страниц на один и тот же рабочий скрипт.
- Не оставляем страницу без пути к коду и без владельца.
