# 4.2 Стандарт структуры скрипта

> Любой ScriptRunner-скрипт должен иметь понятный и повторяемый внутренний каркас.

## Базовая структура

Рекомендуемый порядок блоков:

1. Импорты
2. Константы и настройки
3. Вспомогательные функции
4. Основная функция или основной сценарий
5. Обработка ошибок
6. Логирование результата

## Правила структуры

- конфигурация отделяется от логики;
- вспомогательные функции не должны быть спрятаны внизу без причины;
- основная логика должна читаться сверху вниз;
- if/else-ветки не должны расти бесконтрольно;
- побочные эффекты должны быть видны явно.

## Для разных типов скриптов

- `Listener` — сначала проверка события, потом фильтрация, потом действие;
- `Scheduled Job` — сначала защита от некорректного запуска, потом чтение данных, потом обработка;
- `Workflow Function` — сначала входной контекст, потом проверка условий, потом изменение состояния;
- `REST Endpoint` — сначала валидация запроса, потом основная логика, потом формирование ответа.

## Минимальные skeleton-примеры

### Scheduled Job

```groovy
log.info("GLPI sync started")

def items = loadSourceData()
if (!items) {
    log.warn("GLPI sync skipped: no items to process")
    return
}

def result = syncItems(items)
log.info("GLPI sync finished: processed=${result.processed}, errors=${result.errors}")
```

### Listener

```groovy
if (!isSupportedEvent(event)) {
    return
}

def issue = event.issue
if (!issue) {
    log.warn("Listener skipped: issue is missing")
    return
}

processIssueEvent(issue)
```

### REST Endpoint

```groovy
def payload = parseRequest(request)
if (!payload.isValid()) {
    return badRequest("Invalid payload")
}

def result = service.execute(payload)
return ok(result)
```

## Если скрипт маленький

Не нужно искусственно усложнять код ради идеальной архитектуры. Для небольшого скрипта достаточно:

- явного входа;
- пары понятных вспомогательных функций;
- прозрачной обработки ошибок;
- короткого полезного лога.

## Чего не делаем

- Не прячем основную логику между десятками вспомогательных функций.
- Не держим конфигурацию, бизнес-логику и логирование в одном бесформенном блоке.
- Не допускаем, чтобы основной сценарий читался снизу вверх.
