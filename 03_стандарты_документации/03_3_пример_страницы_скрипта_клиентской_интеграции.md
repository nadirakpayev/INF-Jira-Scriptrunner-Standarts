# 3.3 Пример страницы скрипта: клиентская интеграция

> Ниже приведен пример страницы для клиентского интеграционного сценария на основе логики уровня `Интеграция AMX-Jira`.

## Summary

Скрипт получает список устройств из системы AMX и предоставляет возможность выбора устройства при создании заявки в Jira Service Management. Запускается при обращении к интеграционному сценарию из формы заявки. Основной результат — пользователь выбирает актуальное устройство из внешней системы без ручного ввода.

## Назначение

Скрипт нужен для подключения клиентской системы AMX к процессу создания заявок в Jira и снижения количества ошибок при ручном выборе устройств.

## Тип скрипта и точка запуска

- Тип: `REST Endpoint / Integration Script`
- Запуск: по запросу из формы или связанного сценария интеграции

## Владелец и контекст

- Владелец: команда Jira
- Контекст: клиентский интеграционный скрипт
- Клиент: Azercell
- Критичность: высокая

## Путь к коду

- `scripts/clients/azercell/amx/syncDevicesFromAmx.groovy`

## Входные данные

- параметры запроса из Jira;
- данные API AMX;
- клиентские правила фильтрации и маппинга;
- технические настройки подключения.

## Выход или эффект

- пользователю возвращается список устройств из AMX;
- в Jira используется корректный идентификатор устройства;
- ошибки интеграции фиксируются в логах и не остаются "немыми".

## Зависимости

- внешний API AMX;
- клиентские поля и форма Jira;
- настройки авторизации;
- логгер `com.inforce.logger`.

## Логирование

- `INFO` — старт обращения к AMX и успешное завершение;
- `WARN` — частичная проблема или пустой результат;
- `ERROR` — ошибка запроса к AMX или ошибка обработки ответа.

Примеры:

- `AMX device lookup started: client=Azercell`
- `AMX device lookup finished: devices=24`
- `AMX device lookup failed: step=fetchDevices, client=Azercell`

## Проверка после изменения

1. Открыть форму заявки в Jira.
2. Проверить, что список устройств загружается из AMX.
3. Проверить, что выбранное устройство корректно передается в Jira.
4. Проверить поведение при недоступности AMX и убедиться, что ошибка отражается в логах и диагностируется.

## Риски и ограничения

- зависимость от доступности AMX;
- риск изменения клиентского контракта API;
- сильная привязка к одному клиенту;
- необходимость отдельно проверять поведение после изменений в форме Jira.
