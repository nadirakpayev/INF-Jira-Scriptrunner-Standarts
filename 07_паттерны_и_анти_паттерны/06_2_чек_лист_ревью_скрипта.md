# 6.2 Чек-лист ревью скрипта

> Эта страница нужна для быстрого и практичного ревью ScriptRunner-скрипта без лишней бюрократии.

## Как использовать этот чек-лист

Открывайте его, когда нужно понять:

- можно ли выпускать скрипт;
- чего в нем не хватает;
- где риск для поддержки;
- где решение слишком сложное;
- где локальный костыль выдается за нормальный стандарт.

Это не документ "для галочки". Это короткий рабочий фильтр перед публикацией и при ревью существующего скрипта.

## Блок 1. Понимание назначения

Проверьте:

- понятно ли, что делает скрипт;
- понятно ли, где он запускается;
- понятен ли ожидаемый результат;
- понятно ли, для кого он нужен: общий, клиентский, операционный, миграционный или временный контур.

Красный флаг:

- без автора невозможно за `2-3` минуты понять назначение скрипта.

## Блок 2. Страница и владение

Проверьте:

- есть ли страница в Confluence;
- указан ли владелец;
- указан ли путь к коду;
- есть ли зависимости и шаги проверки;
- есть ли блок `Операционные заметки`.

Красный флаг:

- код есть, а нормальной страницы нет.

## Блок 3. Хранение и границы

Проверьте:

- лежит ли код в правильном контуре;
- не попал ли клиентский код в `shared/`;
- не выдается ли временный скрипт за нормальный рабочий артефакт;
- не размазан ли один сценарий по случайным местам.

Красный флаг:

- переиспользуемый код зависит от клиента, проекта или ручных договоренностей.

## Блок 4. Конфигурация и секреты

Проверьте:

- нет ли в коде токенов, паролей и URL;
- вынесены ли environment-specific параметры;
- вынесены ли клиентские IDs;
- есть ли понятная точка получения конфигурации.

Красный флаг:

- в коде жестко зашиты секреты или критичные параметры среды.

## Блок 5. Структура кода

Проверьте:

- есть ли guard clauses;
- отделены ли transport, mapping, business rules и logging;
- не получился ли скрипт-комбайн;
- есть ли понятные имена методов и переменных;
- нет ли мертвой логики и временных комментариев.

Красный флаг:

- один файл делает все сразу и не делится на понятные шаги.

## Блок 6. Выбор подхода и API

Проверьте:

- не решается ли задача встроенным скриптом или встроенным listener ScriptRunner;
- не лучше ли использовать HAPI;
- если выбран custom path, объяснен ли этот выбор;
- если есть внешний вызов, используется ли supported API.

Красный флаг:

- кастомный скрипт написан там, где был более простой поддерживаемый путь.

## Блок 7. Интеграционная устойчивость

Проверьте, если скрипт интеграционный:

- определен ли источник истины;
- понятен ли ключ сопоставления;
- определены ли `create / update / skip`;
- есть ли timeout;
- есть ли понятная логика повторных попыток;
- различаются ли технические и бизнес-ошибки.

Красный флаг:

- неясно, что будет при повторном запуске или повторной доставке события.

## Блок 8. Работа с Assets и AQL

Проверьте, если скрипт работает с `Assets`:

- ограничен ли AQL по object type;
- используется ли устойчивый атрибут;
- нет ли `anyAttribute` без сильной причины;
- не слишком ли глубокая dot notation;
- понятны ли reference types и условия поиска.

Красный флаг:

- AQL сложный, неименованный и непонятный без автора.

## Блок 9. Логирование и диагностика

Проверьте:

- есть ли понятные `INFO`, `WARN`, `ERROR`;
- видно ли старт и итог сценария;
- видно ли количество обработанных сущностей;
- понятна ли точка отказа;
- можно ли сопровождать скрипт по логам и execution history.

Красный флаг:

- логов много, но по ним нельзя понять, что реально произошло.

## Блок 10. Жизненный цикл и supportability

Проверьте:

- понятны ли шаги проверки после изменения;
- обновлена ли документация;
- есть ли история изменения поведения;
- ясно ли, временный это скрипт или постоянный;
- понятно ли, как его проверять после обновления.

Красный флаг:

- скрипт нельзя безопасно менять без автора.

## Итоговые статусы ревью

### Можно выпускать

Если:

- нет критичных рисков;
- страница заполнена;
- конфигурация и логирование в порядке;
- сценарий поддерживаем без автора.

### Выпускать после точечных правок

Если:

- логика в целом нормальная;
- есть `1-3` конкретных пробела;
- их можно закрыть без пересборки решения.

### Нужна переделка

Если:

- скрипт-комбайн;
- хардкод секретов;
- непонятная модель create/update;
- слабая документация;
- сильная зависимость от автора;
- скрытые риски для production.

## Быстрый yes/no список

- Понятно назначение скрипта?
- Есть страница и владелец?
- Код лежит в правильном месте?
- Нет жестко зашитых секретов?
- Понятна структура кода?
- Подход к API выбран осознанно?
- Для интеграций понятны `create / update / skip`?
- Логи позволяют сопровождать скрипт?
- Есть шаги проверки после изменения?
- Скрипт можно поддерживать без автора?

Если хотя бы на `2-3` пункта ответ `нет`, скрипт не готов как зрелый production-артефакт.
