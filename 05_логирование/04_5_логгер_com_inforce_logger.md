# Логгер com.inforce.logger

> Эта страница фиксирует, как команда использует `com.inforce.logger` в рабочих скриптах.

## Назначение

`com.inforce.logger` используется как единый способ формировать понятные сообщения в логах и поддерживать общий стиль логирования.

## Когда использовать

Используйте этот логгер:

- в production-скриптах;
- в интеграционных сценариях;
- в scheduled jobs;
- в местах, где критична диагностика ошибок.

## Что должно быть единообразным

- стиль сообщений;
- уровень логирования;
- указание ключевой сущности;
- указание шага выполнения;
- указание результата или ошибки.

## Рекомендуемый формат сообщения

Базовый шаблон:

`[операция] + [контекст] + [ключевая сущность] + [шаг] + [итог]`

Например:

- `GLPI sync started: entity=virtualMachine, step=loadSourceData`
- `AMX request updated: client=Azercell, requestId=AMX-4412`
- `Mail handler skipped message: reason=no issue key in subject`

## Практика применения по кейсам

### Кейс: Интеграция GLPI - Jira

Для batch-интеграции логгер должен помогать видеть:

- старт синхронизации;
- тип сущности;
- сколько объектов обработано;
- были ли конфликты данных;
- были ли частичные ошибки.

Хороший результат:

- лог можно использовать и для поддержки, и для последующего разбора импорта.

### Кейс: Интеграция AMX-Jira

Для клиентской интеграции логгер должен помогать видеть:

- к какому клиенту относится сценарий;
- какой шаг выполняется: lookup, create, update;
- какой запрос или сущность обрабатывается;
- где произошла ошибка: авторизация, HTTP, маппинг, listener, behaviour.

### Кейс: Mail Handler for B2B

Для mail-сценария логгер должен помогать видеть:

- найден ли ключ Jira-задачи;
- кто отправитель: внутренний или внешний;
- создан ли комментарий;
- прикреплены ли вложения;
- почему письмо было пропущено.

## Минимальные правила

1. Не используйте логгер как замену структуре кода.
2. Не логируйте чувствительные данные без необходимости.
3. Не смешивайте полезные диагностические сообщения и бессмысленный шум.
4. Если логгер используется как обертка, это должно быть прозрачно для читающего код.

## Хорошо и плохо

Хорошо:

- `GLPI sync finished: entity=software, processed=142, errors=3`
- `AMX lookup failed: client=Azercell, step=authorize, status=401`
- `Mail handler skipped message: reason=no issue key`

Плохо:

- `sync done`
- `error happened`
- `request failed`
- лог на каждом элементе цикла без финального итога

## Когда логгер особенно обязателен

- в scheduled jobs;
- в интеграциях с внешними системами;
- в сценариях с частичными ошибками;
- в клиентских процессах, где ошибка влияет на заявку, объект или синхронизацию.

## Чего не делаем

- Не используем логгер для маскировки плохой структуры кода.
- Не пишем разный стиль сообщений в соседних скриптах одного сценария.
- Не логируем "успех" без указания результата.
- Не оставляем ошибку без контекста шага и сущности.

## Минимальный набор сообщений для production-скрипта

Для рабочего скрипта обычно достаточно такого набора:

1. `INFO` о старте.
2. `INFO` об успешном завершении.
3. `WARN` о частичных отклонениях.
4. `ERROR` о полном сбое.
5. При необходимости `DEBUG` для технической диагностики.
