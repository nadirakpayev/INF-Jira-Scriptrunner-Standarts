# 3.5 Конфигурация, секреты и параметры

> Эта страница задает простой стандарт: что можно хранить в коде, что нужно выносить в параметры, а что вообще нельзя держать в открытом виде.

## Для чего нужна эта страница

Большая часть хрупкости в ScriptRunner-скриптах появляется не из-за Groovy, а из-за хаотичной конфигурации:

- URL зашиты в код;
- токены лежат в тексте страницы;
- field IDs размазаны по нескольким файлам;
- клиентские параметры смешаны с переиспользуемым кодом;
- никто не понимает, что можно безопасно поменять, а что нельзя.

Эта страница нужна, чтобы конфигурация была управляемой.

## Главный принцип

Код описывает логику.
Параметры описывают среду и настройки.
Секреты хранятся только в согласованном защищенном хранилище и никогда не живут в открытом виде в коде или на странице.

## Что такое конфигурация

К конфигурации относятся:

- base URL;
- timeout;
- количество повторных попыток;
- имена и идентификаторы полей;
- object type IDs и schema IDs;
- project keys;
- group names;
- названия integration endpoints;
- флаги включения и отключения функций;
- пути и package names, если они зависят от среды;
- внешние идентификаторы клиентов и систем.

## Что такое секрет

Секретом считаются:

- пароль;
- token;
- API key;
- client secret;
- private key;
- credentials для mail, REST, database, webhook или другой внешней системы.

Секреты не должны:

- попадать в код;
- попадать в Confluence-страницы в открытом виде;
- жить в комментариях;
- жить в истории изменений страницы или скрипта;
- копироваться в несколько мест "для удобства".

## Что можно оставлять в коде

В коде допустимо оставлять:

- локальные технические константы;
- enum-like значения;
- стабильные log prefixes;
- названия внутренних методов и package structure;
- значения, которые не зависят от среды и не считаются чувствительными.

Примеры допустимого:

- `MAX_BATCH_SIZE = 100`;
- `LOG_PREFIX = "[GLPI_SYNC]"`;
- `STATUS_DONE = "Done"` если это часть самой бизнес-логики и не зависит от среды.

## Что не должно быть жестко зашито в код

Нельзя хардкодить в рабочем коде:

- base URL внешней системы;
- credentials;
- tokens и API keys;
- environment-specific project keys, field IDs, objectType IDs;
- customer-specific org IDs, schema IDs, queue IDs;
- e-mail адреса интеграционных учеток, если они могут меняться;
- флаги включения/отключения интеграции.

Если значение может измениться без изменения самой бизнес-логики, это кандидат в параметр.

## Классификация параметров

### 1. Общие параметры

Используются в нескольких скриптах и относятся к общему контуру:

- host систем;
- общие timeout policies;
- общие integration prefixes;
- общие feature flags.

### 2. Клиентские параметры

Относятся только к одному клиенту или проекту:

- customer schema IDs;
- customer field mappings;
- клиентские имена endpoint-ов;
- клиентские project keys.

Такие параметры нельзя маскировать под переиспользуемые константы.

### 3. Секреты

Хранятся только в согласованном защищенном хранилище и по возможности изолированы от обычной конфигурации.

### 4. Временные параметры

Используются только для migration, rollback или временного обхода.

Для них обязательно:

- срок жизни;
- владелец;
- дата пересмотра;
- явная пометка, что параметр временный.

## Рекомендуемая модель хранения

Используем такую схему:

1. В коде есть только имена параметров и логика их использования.
2. Несекретные параметры лежат в одном понятном конфигурационном контуре.
3. Секреты лежат в согласованном защищенном хранилище.
4. На странице в Confluence перечислены имена параметров, но не их секретные значения.

Важно:

- страница документации должна объяснять, что за параметр нужен;
- страница не должна раскрывать чувствительные значения.

## Как документировать параметры на странице скрипта

На странице скрипта должен быть блок:

`Параметры и секреты`

Минимум в нем:

- имя параметра;
- для чего используется;
- обязателен он или нет;
- секрет это или нет;
- кто владеет значением;
- что сломается, если значение неверно.

Хороший формат:

| Имя | Тип | Назначение | Обязательно | Где хранится |
|---|---|---|---|---|
| `GLPI_BASE_URL` | параметр | базовый URL GLPI API | да | согласованный конфигурационный контур |
| `GLPI_API_TOKEN` | секрет | токен доступа к GLPI | да | согласованное защищенное хранилище |
| `CMDB_SYNC_TIMEOUT_MS` | параметр | таймаут внешнего вызова | да | согласованный конфигурационный контур |

## Как оформлять доступ к параметрам в коде

Хороший подход:

- получить параметры в одном месте;
- проверить обязательные значения в начале сценария;
- не размазывать чтение параметров по всему скрипту;
- валидировать тип и наличие до основной логики.

Плохо:

- читать конфигурацию в пяти разных функциях;
- использовать секрет прямо внутри запроса без центральной точки получения;
- смешивать бизнес-логику с извлечением параметров.

## Когда нужна отдельная config-модель

Если интеграция:

- использует больше `3-5` параметров;
- работает с несколькими системами;
- имеет клиентский mapping;
- требует feature flags или временных режимов,

стоит вынести параметры в отдельную config-модель или хотя бы в один явный конфигурационный блок.

## Ротация секретов

Стандарт требует, чтобы:

- секрет можно было заменить без переписывания бизнес-логики;
- ротация секрета не требовала поиска по нескольким файлам;
- после ротации был понятный шаг проверки;
- владелец секрета был известен.

Если секрет заменить трудно, конфигурационная модель плохая.

## Минимум без бюрократии

Для обычного рабочего скрипта достаточно:

1. Все чувствительные значения вынесены из кода.
2. Параметры среды не зашиты в код.
3. На странице перечислены имена параметров.
4. В коде есть одна понятная точка получения конфигурации.
5. При ошибке конфигурации лог показывает, какой параметр отсутствует или некорректен, но не печатает секрет.

## Чего не делаем

- Не кладем токены и пароли в код.
- Не вставляем секреты в Confluence в открытом виде.
- Не размазываем field IDs и URLs по нескольким скриптам.
- Не используем жестко заданные клиентские значения в переиспользуемых модулях.
- Не печатаем секреты в лог при ошибке.
- Не храним временный параметр без срока пересмотра.

## Короткий чек-лист перед публикацией

- Секреты вынесены из кода.
- Environment-specific параметры вынесены из кода.
- Клиентские параметры отделены от общего переиспользуемого кода.
- На странице есть список параметров по именам.
- Конфигурация валидируется в начале сценария.
- Ошибки конфигурации видны в логах безопасным образом.
