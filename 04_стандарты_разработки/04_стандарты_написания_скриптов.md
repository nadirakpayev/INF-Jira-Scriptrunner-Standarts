# 4. Стандарты написания скриптов

> Эта страница задает обязательный минимум для любого нового или изменяемого ScriptRunner-скрипта.

## Обязательные правила

1. Код должен быть читаемым без автора.
2. В одном скрипте должна быть одна основная ответственность.
3. Скрытая бизнес-магия запрещена.
4. Значимые решения должны быть названы явно.
5. Ошибки должны обрабатываться предсказуемо.
6. Логирование должно быть встроено осознанно.
7. Повторяющуюся логику нужно выносить, но без чрезмерного дробления.

## Что рекомендуется

- Использовать guard clauses в начале сценария.
- Отделять чтение данных, бизнес-правила и запись результата.
- Давать функциям и переменным имена, понятные без открытия Confluence.
- Делать ключевые шаги сценария видимыми в коде и логах.
- Писать так, чтобы скрипт можно было безопасно доработать через несколько месяцев.

## HAPI и встроенные механизмы ScriptRunner

Для типовых операций в Jira рекомендуемый порядок такой:

1. Сначала проверить, нет ли подходящего встроенного скрипта или встроенного listener ScriptRunner.
2. Если нужен код, сначала проверить, можно ли решить задачу через HAPI.
3. Классический Jira API использовать, если встроенные механизмы и HAPI не покрывают сценарий или нужен более точный контроль.

Это уменьшает boilerplate и снижает порог входа для поддержки.

## Статическая проверка типов

Static Type Checking полезен как часть ежедневной разработки:

- помогает ловить проблемы до запуска;
- делает код понятнее;
- заставляет раньше замечать deprecated API.

При этом STC не является абсолютной истиной:

- в динамических сценариях и тестах возможны false positives;
- такие предупреждения не нужно слепо игнорировать, но и не нужно считать автоматической ошибкой runtime.

Практическое правило:

- если типизация помогает читаемости и STC, используйте ее;
- если сценарий динамический, фиксируйте это осознанно, а не случайно.

## Что считается плохим кодом

- длинные монолитные блоки без структуры;
- неочевидные сокращения;
- смешение бизнес-логики, интеграции и форматирования в одном месте;
- дублирование одной и той же логики в нескольких скриптах;
- отсутствие проверок на ошибочные состояния;
- отсутствие понятных логов.

## Что считается хорошим кодом

- короткие понятные блоки;
- явные названия функций и переменных;
- минимальный побочный эффект;
- понятные шаги выполнения;
- ясная обработка ошибок;
- предсказуемый результат.

## Как выглядит минимально хороший скрипт

Минимально хороший скрипт:

- сначала отсекает неподходящие условия;
- явно показывает вход в основной сценарий;
- не прячет важные значения в "магических" константах;
- не смешивает все в один бесформенный блок;
- пишет полезные логи;
- позволяет другому человеку понять, как его проверить.

## Чего не делаем

- Не строим "умную" архитектуру, если задача решается простым читаемым скриптом.
- Не выносим код в `shared/` только ради красоты.
- Не оставляем критичную бизнес-логику без документации.
- Не пишем код, который можно безопасно менять только автору.

## Пример из практики

В интеграционном сценарии уровня `Интеграция GLPI - Jira` хороший код выглядит так:

- отдельно читаются входные данные;
- отдельно выполняется маппинг;
- отдельно создаются или обновляются объекты;
- в логах видно, сколько объектов обработано и сколько завершилось с ошибкой.

Плохой вариант для такого сценария — один длинный скрипт, где чтение API, преобразование, бизнес-правила и запись в Assets смешаны в одном блоке.
