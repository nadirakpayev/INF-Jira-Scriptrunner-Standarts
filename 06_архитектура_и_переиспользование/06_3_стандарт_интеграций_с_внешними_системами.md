# 6.3 Стандарт интеграций с внешними системами

> Эта страница задает практический стандарт для интеграций, которые обмениваются данными между Jira и внешними системами.

## Для чего нужна эта страница

Сценарии уровня `GLPI`, `AMX`, `CMDB`, `mail-to-ticket`, внешние каталоги и клиентские API чаще всего ломаются не из-за одного метода в коде, а из-за слабой интеграционной модели:

- непонятен источник истины;
- смешаны transport, auth, mapping и business rules;
- не определено, что делать при частичной ошибке;
- неясно, когда делать `create`, `update`, `skip` или повторную попытку;
- в коде зашиты параметры, которые нельзя безопасно менять.

Эта страница нужна, чтобы такие интеграции были не просто рабочими, а поддерживаемыми.

## Когда применять этот стандарт

Стандарт обязателен, если скрипт:

- вызывает внешний API;
- принимает события или данные от внешней системы;
- публикует данные во внешнюю систему;
- синхронизирует Jira с внешним справочником или CMDB;
- использует custom REST endpoint как вход в Jira;
- зависит от сетевых вызовов, токенов, URL, внешних идентификаторов или логики повторных попыток.

## Главный принцип

Сначала проектируем контракт интеграции, потом пишем код.

Правильный порядок такой:

1. Определяем источник истины.
2. Определяем поток данных.
3. Определяем контракт внешней системы.
4. Определяем правила `create / update / skip / повторная попытка / fail`.
5. Определяем логирование и способ поддержки.
6. Только после этого пишем скрипт.

## Допустимые паттерны интеграции

### 1. Outbound pull / scheduled sync

Jira или ScriptRunner по расписанию запрашивает данные из внешней системы.

Подходит для:

- каталогов;
- CMDB/Assets-синхронизаций;
- пакетной сверки данных;
- процессов, где мгновенная реакция не обязательна.

### 2. Inbound push / webhook / custom REST endpoint

Внешняя система сама вызывает Jira через endpoint.

Подходит для:

- уведомлений;
- событийных интеграций;
- сценариев, где внешняя система инициирует изменение.

### 3. Hybrid pattern

Часть логики работает по push, часть по scheduled reconciliation.

Подходит для:

- критичных процессов, где event-driven сценарий удобен, но нужна сверка и восстановление консистентности.

## Что должно быть определено до начала разработки

До начала разработки должно быть ясно:

- какая система является источником истины;
- какие сущности передаются;
- какие поля обязательны;
- какие идентификаторы устойчивы;
- что считается успешной обработкой;
- что считается временной ошибкой;
- что считается бизнес-конфликтом;
- что считается полным сбоем;
- нужна ли повторная попытка (`retry`);
- нужен ли пробный прогон (`dry-run`);
- кто владелец интеграции;
- кто разбирает проблему первым: разработка, поддержка или владелец процесса.

Если на эти вопросы нет ответа, интеграция еще не готова к реализации.

## Поддерживаемый путь для транспорта и API

Для внешних интеграций используем поддерживаемые интерфейсы в таком порядке:

1. Официальный `REST API` системы.
2. Поддерживаемый `custom REST endpoint` ScriptRunner, если Jira должна принимать внешние вызовы.
3. Локальный низкоуровневый API только если поддерживаемый путь не покрывает сценарий.

Не делаем так:

- не строим интеграцию через прямой доступ к базе данных;
- не используем неофициальные обходы, если есть поддерживаемый REST;
- не открываем custom endpoint без явной модели доступа.

## Правила для custom REST endpoints

Если интеграция использует custom REST endpoint ScriptRunner, обязательно:

- явно задавать `httpMethod`;
- явно задавать `groups`;
- не оставлять endpoint "по умолчанию";
- документировать, кто имеет право вызывать endpoint;
- документировать формат входного payload и expected response.

Это критично, потому что по официальной документации Adaptavist, если у endpoint не указан `groups`, он будет доступен неаутентифицированным пользователям.

## Разделяйте роли в интеграции

В одном интеграционном сценарии должны быть логически разделены:

- transport;
- auth;
- parsing;
- validation;
- mapping;
- business rules;
- persistence;
- logging.

Пример хорошего подхода по мотивам `AMX -> Jira`:

- отдельно HTTP-вызовы;
- отдельно авторизация;
- отдельно получение и преобразование данных;
- отдельно listeners / behaviours / update flow.

Плохо, когда все это слито в один файл.

## Источник истины и конфликт данных

Для каждой интеграции надо явно выбрать:

- кто главный по объекту или полю;
- какие поля можно обновлять только из Jira;
- какие поля можно обновлять только извне;
- какие поля конфликтуют;
- что делать, если входящие данные противоречат текущему состоянию.

Если не определить источник истины, интеграция очень быстро превращается в хаотичный перезаписывающий сценарий.

## Правило create / update / skip / delete

Для каждой сущности заранее определяем:

- как ищется существующая запись;
- по какому ключу выполняется `match`;
- когда выполняется `create`;
- когда выполняется `update`;
- когда выполняется `skip`;
- когда допустим `delete`, а когда только `deactivate` или `archive`.

Нельзя оставлять это на уровне "если не нашли, наверное создаем".

## Idempotency

Интеграция должна быть максимально идемпотентной.

Это значит:

- повторный запуск не должен создавать дубликаты;
- повторная доставка одного и того же события не должна ломать данные;
- повторная попытка не должна ухудшать состояние;
- запись должна быть привязана к устойчивому идентификатору.

Для batch-сценариев и webhook-сценариев это обязательное правило.

## Повторные попытки и таймауты

У каждой внешней интеграции должна быть понятная стратегия:

- какой timeout считаем нормальным;
- сколько раз делаем повторную попытку;
- при каких ошибках повторная попытка вообще допустима;
- при каких ошибках надо сразу завершать сценарий;
- нужен ли backoff.

Хороший минимум:

- timeout не оставляется "как получится";
- повторная попытка не делается бесконечно;
- повторная попытка не маскирует бизнес-ошибку;
- итоговый лог показывает, была ли повторная попытка и чем она закончилась.

## Частичные ошибки

Для batch и sync-сценариев частичная ошибка считается нормальным режимом, а не исключением.

Интеграция должна различать:

- полный успех;
- частичный успех;
- технический сбой;
- бизнес-конфликт.

Пример по мотивам `GLPI -> Jira Assets`:

- одна невалидная сущность не должна останавливать весь пакет без причины;
- в логах должен быть итог по успешным, пропущенным и ошибочным записям.

## Пробный прогон и безопасный режим

Если интеграция:

- массово меняет данные;
- создает или обновляет объекты пакетно;
- использует новую схему маппинга;
- впервые запускается после изменения,

для нее желательно иметь пробный прогон (`dry-run`), `preview` или другой безопасный режим проверки.

Это особенно полезно для:

- CMDB/Assets-синхронизаций;
- миграционных прогонов;
- крупных customer-specific обновлений.

## Что обязательно логировать

Для каждой интеграции в логах должно быть видно:

- старт и завершение сценария;
- источник или тип запуска;
- количество обработанных сущностей;
- количество созданных, обновленных, пропущенных и ошибочных объектов;
- события повторных попыток и таймаутов;
- идентификатор или тип сущности, если произошла ошибка;
- итоговый статус.

Если лог не отвечает на вопрос "что произошло и сколько данных реально изменилось", он слабый.

## Что должно быть на странице интеграции в Confluence

На странице интеграции обязательно должны быть:

- цель интеграции;
- источник истины;
- направление потока данных;
- ключевые сущности и атрибуты;
- схема `create / update / skip`;
- описание политики повторных попыток и таймаутов;
- список параметров и секретов по именам;
- точки запуска;
- зависимости;
- блок `Операционные заметки`;
- шаги проверки после изменения;
- известные ограничения.

## Минимум без бюрократии

Для обычной рабочей интеграции достаточно этого минимума:

1. Понятно, кто источник истины.
2. Понятно, как ищется существующая запись.
3. Понятно, где transport, а где business rules.
4. Есть таймаут и понятная стратегия повторных попыток.
5. Логи показывают итог работы.
6. Параметры и секреты не зашиты в код.
7. На странице интеграции есть шаги проверки.

## Чего не делаем

- Не смешиваем HTTP, auth, mapping и бизнес-логику в один непрозрачный блок.
- Не создаем custom REST endpoint без явной модели доступа.
- Не держим интеграцию на жестко заданных URL, токенах и идентификаторах среды.
- Не делаем batch-сценарий, который "роняет все" из-за одной записи без особой причины.
- Не используем внешний API без timeout и без понятного failure-path.
- Не делаем повторную попытку там, где ошибка на самом деле бизнесовая, а не техническая.

## Короткий чек-лист перед публикацией

- Источник истины определен.
- Ключ сопоставления (`match-key`) определен.
- Правила `create / update / skip` определены.
- таймаут и повторные попытки определены.
- endpoint защищен, если он есть.
- секреты не лежат в коде.
- логирование показывает итог.
- страница интеграции заполнена.
