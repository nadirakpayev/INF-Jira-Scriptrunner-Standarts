# 4.5 HAPI и классический API

> Эта страница помогает выбрать самый простой и поддерживаемый способ реализации задачи в ScriptRunner.

## Главный принцип

Мы не должны автоматически писать код на низкоуровневом Jira API, если задачу можно решить проще.

Рекомендуемый порядок выбора такой:

1. Проверить, нет ли встроенного скрипта или встроенного listener ScriptRunner.
2. Если нужен код внутри Jira, сначала рассмотреть `HAPI`.
3. Если задача интеграционная и работает с внешней системой, сначала рассмотреть официальный `REST API`.
4. Классический Jira API использовать, если HAPI или встроенный подход не покрывают сценарий, либо нужен более точный контроль.

Это уменьшает boilerplate, снижает порог входа и делает код понятнее для поддержки.

## Что такое HAPI

`HAPI` в ScriptRunner — это упрощенный API для типовых задач в Jira.

Его сильные стороны:

- меньше шаблонного кода;
- ниже порог входа;
- проще читать и менять;
- хорошие completions в редакторе ScriptRunner;
- подходит и новичку, и опытному разработчику.

Практический вывод для нашего стандарта:

- если задача типовая и находится внутри Jira, `HAPI` должен быть первым кандидатом.

## Когда выбирать HAPI

HAPI обычно лучший выбор, если нужно:

- создать или обновить задачу;
- работать с полями;
- искать задачи;
- работать со связями между задачами;
- добавлять watchers, comments и другие типовые Jira-операции;
- быстро реализовать понятную automation-логику без тяжелого boilerplate.

### Пример подходящего HAPI-сценария

Подходящие задачи:

- listener, который добавляет watchers;
- скрипт, который выставляет поле при определенном условии;
- автоматизация по request type;
- простая post-function логика.

Для таких задач HAPI чаще всего лучше классического Jira API, потому что код получается короче и прозрачнее.

## Когда HAPI недостаточен

HAPI не обязан покрывать все сценарии.

Классический Jira API может понадобиться, если:

- нужен более низкий уровень контроля;
- работаете с API, которое HAPI пока не покрывает;
- сценарий опирается на специфичные Jira / JSM компоненты;
- нужно тонко управлять event context, service utils или внутренними объектами платформы;
- вы уже находитесь в контуре plugin-like логики, где требуется работа с более низким уровнем Jira API.

## Когда выбирать классический Jira API

Классический Jira API — это не "плохой выбор". Это инструмент для более сложных или специальных сценариев.

Его стоит использовать, если:

- HAPI не покрывает нужную операцию;
- нужен точный контроль над компонентами Jira;
- нужно использовать стабильные публичные Java API Jira;
- решение невозможно адекватно выразить через HAPI без обходных конструкций.

Практическое правило:

- классический API берем не "по привычке", а по явной причине.

Эта причина должна быть понятна из кода или документации страницы скрипта.

## Когда выбирать REST API

Если скрипт взаимодействует с внешней системой, первым выбором обычно должен быть официальный `REST API`, а не внутренние обходы.

Это особенно важно для сценариев уровня:

- `Интеграция GLPI - Jira`;
- `Интеграция AMX-Jira`;
- другие клиентские и инфраструктурные интеграции.

### Почему это важно

- REST API поддерживается официально;
- контракт понятнее и стабильнее;
- интеграцию проще объяснить и тестировать;
- меньше риск завязаться на внутренности платформы или базы данных.

## Простое правило выбора

| Сценарий | Первый выбор | Почему |
|---|---|---|
| Разовая административная задача, уже покрытая ScriptRunner | Встроенный скрипт ScriptRunner | Быстрее, проще, меньше кастомного кода |
| Типовая Jira-логика внутри ScriptRunner | HAPI | Меньше boilerplate, выше читаемость |
| Интеграция с внешней системой | REST API | Поддерживаемый контракт и понятная интеграционная модель |
| Нестандартный или низкоуровневый Jira-сценарий | Классический Jira API | Нужен более точный контроль |

## Как это применять в нашей практике

### Сценарий 1. Простая внутренняя автоматизация

Если нужно:

- обновить поле;
- создать связанную задачу;
- добавить watcher;
- обработать типовой issue event;

первым выбором должен быть `HAPI`.

### Сценарий 2. Клиентская интеграция

Если нужно:

- сходить во внешний API;
- получить данные;
- преобразовать их;
- обновить Jira или JSM;

архитектура обычно должна выглядеть так:

- внешний контракт через `REST API`;
- внутренняя Jira-логика через `HAPI` или классический Jira API, в зависимости от покрытия.

### Сценарий 3. Специфичный event-driven код

Если работаете со сценарием уровня `Mail Handler for B2B`, HAPI может быть не основным инструментом, потому что там важны специализированные Jira/JSM компоненты и обработка почтового контекста.

В таком случае допустимо использовать классический API, но:

- причина должна быть ясна;
- логика должна оставаться читаемой;
- страница документации должна объяснять, почему выбран именно такой уровень API.

## Что проверять перед выбором классического API

Перед тем как писать код на классическом Jira API, ответьте на вопросы:

1. Нет ли встроенного скрипта или встроенного listener ScriptRunner для этой задачи?
2. Нельзя ли решить это через HAPI без искусственных обходов?
3. Если это интеграция, используем ли мы официальный REST API внешней системы?
4. Если берем Java API Jira, используем ли мы публичный и поддерживаемый API?
5. Сможет ли другой разработчик понять этот выбор через 3 месяца?

Если на первые два вопроса честный ответ "да, можно проще", классический API не нужен.

## Анти-паттерны

### Анти-паттерн 1. Низкоуровневый API по привычке

Признаки:

- даже простое обновление поля делается через тяжелый boilerplate;
- код длинный, а полезной логики мало;
- новый разработчик не понимает, зачем столько технических слоев.

### Анти-паттерн 2. Внешняя интеграция через неподдерживаемые обходы

Признаки:

- используются внутренние детали вместо официального REST-контракта;
- в коде появляются нестабильные зависимости;
- интеграцию сложно тестировать и поддерживать.

### Анти-паттерн 3. HAPI там, где он уже не помогает

Признаки:

- код на HAPI обрастает обходными конструкциями;
- ради HAPI скрывается важная техническая сложность;
- итоговый скрипт становится менее понятным, чем обычный классический API.

Практический вывод:

- HAPI — первый выбор, но не религия.

## Что писать на странице скрипта

Если выбор неочевиден, на странице скрипта нужно фиксировать:

- какой уровень API используется;
- почему выбран именно он;
- какие ограничения у этого выбора;
- что может потребоваться при будущем рефакторинге или апгрейде.

Особенно это важно для:

- клиентских интеграций;
- mail handlers;
- сложных listeners;
- сценариев с глубокой связью с JSM.

## Что добавляем в наш стандарт как правило

1. Для типовых Jira-операций сначала рассматриваем `HAPI`.
2. Для внешних интеграций сначала рассматриваем официальный `REST API`.
3. Классический Jira API используем по явной причине.
4. Если задача уже покрыта встроенным скриптом ScriptRunner, не пишем кастомный код без необходимости.
5. Выбор уровня API должен упрощать поддержку, а не усложнять ее.
