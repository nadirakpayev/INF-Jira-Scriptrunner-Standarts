# Внешние источники для усиления стандарта разработки на ScriptRunner

> Эта страница собирает официальные внешние ресурсы, которые реально полезны для усиления нашего стандарта. Здесь важны не просто ссылки, а вывод: что именно из этих источников стоит встроить в стандарт и в какую страницу.

## Общий вывод

Внешние официальные источники подтверждают, что наш стандарт движется в правильную сторону, но его можно усилить еще в шести местах:

1. Явно зафиксировать `HAPI-first` подход для типовых задач в Jira.
2. Добавить правило использовать `Script Roots` как основной способ хранения и структурирования кода.
3. Добавить практику `Static Type Checking` как часть разработки, но с учетом ложных срабатываний.
4. Добавить блок про `Spock` и `Test Runner` для критичных скриптов и интеграций.
5. Добавить правило использовать `Script Registry`, `Execution History` и встроенное логирование ScriptRunner как operational part стандарта.
6. Усилить интеграционные страницы правилами для `REST API`, `AQL` и осторожного использования Groovy DSL/closures.

Ниже источники и конкретные действия.

## 1. Adaptavist: Script Roots

Источник:

- [Adaptavist: Script Roots](https://docs.adaptavist.com/sr4js/latest/best-practices/write-code/script-roots)

Что важно:

- ScriptRunner автоматически сканирует script roots.
- По умолчанию создается директория `Scripts` под Jira home.
- Это логичное место для кода, потому что оно сохраняется при апгрейдах Jira и доступно всем нодам в кластере.
- Utility-классы должны иметь package name, совпадающий со структурой директорий.
- Adaptavist отдельно предупреждает про нюанс recompilation: если изменился dependency class, иногда нужно слегка обновить dependent script, чтобы триггернуть перекомпиляцию.

Что добавить в наш стандарт:

- усилить правило, что рабочий код живет внутри script roots;
- явно запретить хранение производственного кода вне script roots без особой причины;
- добавить правило по package names для helper-классов;
- добавить operational note про recompilation зависимостей.

- правило `Низкоуровневый Jira API используем, когда HAPI не покрывает сценарий или нужен более точный контроль`;
- короткую таблицу `HAPI vs classic API`.

## 3. Adaptavist: Static Type Checking

Источник:

- [Adaptavist: Static Type Checking](https://docs.adaptavist.com/sr4js/latest/best-practices/write-code/static-type-checking)

Что важно:

- STC показывает проблемы в коде на этапе написания, а не только при запуске.
- Adaptavist отдельно подчеркивает, что скрипты в runtime все равно компилируются динамически.
- Есть ограничения: builders, closures, Spock DSL могут давать false positives.
- Для классов и closures полезно явно задавать типы, чтобы помочь STC.
- Adaptavist рекомендует учитывать deprecation warnings, потому что deprecated API чаще всего вылетают на major upgrade.

Что добавить в наш стандарт:

- правило `STC используем как обязательный feedback loop при разработке`;
- правило `ложные STC-ошибки не игнорируем автоматически, а сначала классифицируем`;
- правило `closure parameters и service fields типизируем, если это улучшает STC и читаемость`;
- правило `deprecation warnings исправляем до апгрейда`.

## 4. Adaptavist: Write and Run Tests

Источник:

- [Adaptavist: Write and Run Tests](https://docs.adaptavist.com/sr4js/latest/best-practices/write-and-run-tests)

Что важно:

- Adaptavist рекомендует использовать тестовую инстанцию, а не production.
- Поддерживаемый testing framework — `Spock`.
- Для ScriptRunner доступны интеграционные тесты через `Test Runner`.
- Для тяжелых и критичных сценариев тесты особенно полезны после изменений и апгрейдов.
- Есть понятный test-driven workflow: тест -> запуск -> реализация -> рефакторинг -> review -> повторная проверка.

Что добавить в наш стандарт:

- правило `критичные интеграции, scheduled jobs и reusable-модули должны иметь тестовый сценарий`;
- минимум не обязательно делать для всех скриптов, но для high-risk сценариев тестирование должно стать частью стандарта;
- добавить шаблон `что тестировать` для listener / scheduled job / REST endpoint.

## 5. Adaptavist: Script Registry, Execution History, Logging

Источники:

- [Adaptavist: Script Registry](https://docs.adaptavist.com/sr4js/latest/features/script-registry)
- [Adaptavist: Execution History](https://docs.adaptavist.com/sr4js/8.9.0/best-practices/logging/execution-history)
- [Adaptavist: Logging](https://docs.adaptavist.com/sr4js/9.7.0/best-practices/logging)
- [Adaptavist: Advanced Logging](https://docs.adaptavist.com/sr4js/8.8.1/best-practices/logging/advanced-logging)
- [Adaptavist: Audit Logging](https://docs.adaptavist.com/sr4js/latest/best-practices/logging/audit-logging)

Что важно:

- Script Registry помогает видеть type-checking errors, deprecated API и экспортировать scripts/configurations.
- Adaptavist рекомендует использовать Script Registry на staging после апгрейдов.
- Execution History позволяет видеть длительность, сбои и деградацию производительности.
- Встроенный `log` уже инжектится в inline scripts.
- Для Jira 9.5+ важно учитывать переход на Log4j2.
- Audit logging полезен для отслеживания конфигурационных изменений ScriptRunner.

Что добавить в наш стандарт:

- правило `после апгрейда проверяем Script Registry на staging`;
- правило `для критичных job/listener/rest-сценариев смотрим Execution History`;
- правило `у production-скриптов должен быть понятный operational path: где смотреть logs, execution history и audit trace`;
- правило `если нужен отдельный logger category, это должно быть осознанное решение`.

## 7. Groovy Official: style guide and closures

Источники:

- [Groovy Style Guide](https://docs.groovy-lang.org/latest/html/documentation/style-guide.html)
- [Groovy Closures](https://docs.groovy-lang.org/latest/html/documentation/core-closures.html)

Что важно:

- semicolons в Groovy обычно не нужны;
- не стоит смешивать `def` и явный тип в одной декларации;
- `public` по умолчанию можно не писать;
- `with()` и `tap()` полезны для повторяющихся операций над одним объектом;
- `Groovy Truth`, safe navigation `?.`, Elvis `?:` и `assert` помогают писать более короткий и выразительный код;
- closures в Groovy имеют `delegate` и `resolveStrategy`, и это мощно, но может порождать неочевидное поведение.

Что добавить в наш стандарт:

- усилить block по idiomatic Groovy;
- прямо разрешить `?.` и `?:`, если это делает код понятнее;
- не поощрять "умные" Groovy DSL-приемы без необходимости;
- добавить правило `если closure зависит от delegate magic, это должно быть оправдано и понятно`.

## 8. Atlassian Official: REST APIs and Java APIs

Источники:

- [Atlassian: Jira REST APIs](https://developer.atlassian.com/server/jira/platform/jira-rest-apis-7372883/)
- [Atlassian: Jira Java APIs](https://developer.atlassian.com/server/jira/platform/java-apis/)

Что важно:

- Atlassian прямо указывает, что для удаленного взаимодействия поддерживаются REST APIs.
- REST API использует стандартную HTTP-модель и JSON.
- Java API нужен скорее для plugin-level сценариев и более глубоких расширений, чем для обычного внешнего интеграционного скрипта.

Что добавить в наш стандарт:

- правило `для внешних интеграций сначала рассматриваем официальный REST API`;
- правило `не опираться на неофициальные или DB-like обходы, если есть поддерживаемый REST`;
- правило `Java API и более низкий уровень — только если REST/HAPI не покрывают задачу`.

## Финальный вывод

Лучшие внешние источники не требуют радикально менять наш стандарт. Они подтверждают его логику, но подсказывают, где добавить инженерную зрелость:

- меньше "магии" и больше supported paths;
- меньше boilerplate через HAPI;
- больше operational visibility через Script Registry / Execution History;
- больше надежности через STC и тесты;
- больше практической пользы через REST/AQL/HAPI-примеры.

Это делает стандарт не только удобным для чтения, но и ближе к реальной эксплуатации ScriptRunner в production-среде.
