# 3.4 Правила версионирования через Confluence

> Эта страница задает единые правила фиксации изменений, если Confluence используется как основной рабочий слой документации.

## Что считаем версией

Версией считается любое изменение, которое влияет на:

- поведение скрипта;
- входные данные;
- выходные данные;
- точки запуска;
- зависимости;
- риски;
- порядок поддержки.

## Обязательные статусы страницы

- `Draft` — страница в работе;
- `Review` — страница и код готовы к проверке;
- `Active` — рабочая актуальная версия;
- `Deprecated` — использовать больше не нужно;
- `Archived` — исторический материал.

## Обязательные правила

1. У страницы должен быть один актуальный статус.
2. Изменение поведения скрипта без изменения страницы запрещено.
3. Если меняется контракт или логика, это должно быть отражено в истории изменений.
4. Deprecated-скрипт должен быть явно помечен и не должен выглядеть как рабочий.
5. Архивный материал не должен конкурировать с актуальным.

## Рекомендуемый жизненный цикл страницы

Обычный маршрут страницы такой:

`Draft -> Review -> Active -> Deprecated -> Archived`

Расшифровка:

- `Draft` — скрипт или страница еще собираются;
- `Review` — код и документация готовы к проверке;
- `Active` — рабочая версия, на которую можно ссылаться как на актуальную;
- `Deprecated` — решение еще видно для истории, но использовать его больше не нужно;
- `Archived` — только исторический материал, не для рабочей навигации.

## Что писать в истории изменений

Минимум:

- дата;
- автор изменения;
- что изменили;
- зачем изменили;
- как проверить, что изменение работает.

## Copy-ready шаблон истории изменений

```markdown
## История изменений

| Дата | Автор | Что изменили | Зачем изменили | Как проверить |
|---|---|---|---|---|
| 2026-04-19 | Имя Фамилия | Добавлен retry при ошибке внешнего API | Снизить количество ручных перезапусков | Запустить на тестовом наборе данных и проверить повторную обработку |
```

## Когда достаточно короткой записи

Короткой записи в истории изменений достаточно, если:

- менялось описание страницы без изменения поведения;
- уточнялись шаги проверки;
- актуализировались ссылки или владелец;
- исправлялась неточность в документации.

## Когда нужна полноценная запись

Полноценная запись обязательна, если меняется:

- логика скрипта;
- контракт интеграции;
- точка запуска;
- порядок обработки ошибок;
- состав зависимостей;
- набор шагов проверки;
- статус страницы.

## Когда страница переводится в Deprecated

- скрипт больше не должен использоваться;
- есть новый заменяющий скрипт;
- контекст больше не актуален;
- логика перенесена в другой слой.

## Пример жизненного цикла на практике

### Active -> Deprecated

Скрипт синхронизации устройств для клиента заменили новым вариантом с другим контрактом. Старая страница не удаляется сразу, а переводится в `Deprecated`, чтобы:

- было понятно, что раньше работало;
- можно было найти старую логику;
- поддержка не перепутала старый и новый сценарий.

На странице обязательно должно быть:

- явное указание, что страница устарела;
- ссылка на новый скрипт или новую страницу;
- дата вывода из активного использования.

### Deprecated -> Archived

Если скрипт больше не нужен, не используется и больше не участвует в поддержке, страницу можно перевести в `Archived`. В архиве материал остается только как источник истории.

## Чего не делаем

- Не используем `Draft` для давно работающего решения.
- Не оставляем страницу `Active`, если код уже заменен.
- Не архивируем страницу сразу, если по ней еще может понадобиться поддержка или разбор истории.
- Не пишем в истории изменений фразы вроде `fixes`, `update`, `minor changes` без пояснения.
