# 1.3 Когда не нужен кастомный скрипт

> Эта страница нужна, чтобы не писать Groovy без необходимости. Хороший стандарт разработки умеет не только описывать, как писать код, но и заранее отсекать случаи, где код вообще не нужен.

## Главный принцип

Кастомный скрипт не должен быть первым выбором по умолчанию.

Перед тем как писать новый ScriptRunner-скрипт, мы проверяем:

1. Можно ли решить задачу штатной конфигурацией Jira или JSM.
2. Можно ли решить задачу встроенным скриптом ScriptRunner.
3. Можно ли решить задачу встроенным listener ScriptRunner.
4. Можно ли решить задачу через HAPI без тяжелого кастомного слоя.
5. Нужен ли вообще новый код, или достаточно доработать существующий артефакт.

Если задачу можно решить проще и надежнее без нового кастомного Groovy, так и нужно делать.

## Когда кастомный скрипт точно не нужен

### 1. Задача уже решается штатной конфигурацией

Код чаще всего не нужен, если вопрос закрывается через:

- workflow configuration;
- permissions;
- screens;
- field configuration;
- request types;
- SLA settings;
- notification schemes;
- стандартные Jira/JSM механизмы.

Практическое правило:

- если поведение можно собрать из штатной конфигурации без ухудшения поддержки, скрипт не нужен.

### 2. Подходит встроенный скрипт ScriptRunner

Встроенные скрипты ScriptRunner существуют именно для того, чтобы не писать Groovy с нуля для типовых административных и разовых задач.

Это особенно уместно, если задача:

- административная;
- разовая или редкая;
- не несет уникальной бизнес-логики;
- уже покрыта готовым встроенным инструментом.

Примеры задач такого типа:

- массовое исправление полей;
- перенос ownership;
- просмотр scheduled jobs;
- просмотр Script Registry;
- housekeeping и operational scripts.

### 3. Подходит встроенный listener ScriptRunner

Если нужен событийный сценарий, сначала проверяем встроенные listeners ScriptRunner.

Это разумно, когда задача типовая:

- добавить watcher;
- создать sub-task;
- fast-track transition;
- отправить email;
- отправить Slack message;
- notifier на execution failure.

Если встроенный listener закрывает сценарий, писать собственный listener не нужно.

### 4. Задача типовая и решается через HAPI

Если внутри Jira нужно сделать типовую операцию:

- изменить задачу;
- обновить поле;
- добавить watcher;
- найти связанные issues;
- обработать простой event;

то сначала рассматриваем `HAPI`, а не тяжелый custom code на низкоуровневом API.

Технически это уже код, но это не тот уровень кастомизации, который требует сложной собственной архитектуры.

## Когда не надо создавать новый скрипт, но надо доработать существующий

Новый скрипт не нужен, если:

- задача расширяет уже существующий рабочий сценарий;
- логика относится к тому же бизнес-процессу;
- можно безопасно доработать текущий код без искусственного дробления;
- новый файл появится только потому, что никто не хочет разбираться в существующем.

Практическое правило:

- сначала проверяем, не является ли запрос просто изменением существующего артефакта.

## Когда кастомный скрипт все же оправдан

Кастомный скрипт оправдан, если одновременно выполняется хотя бы часть этих условий:

- логика реально уникальная;
- встроенный скрипт ScriptRunner не покрывает сценарий;
- встроенный listener ScriptRunner не покрывает сценарий;
- штатной конфигурации недостаточно;
- нужен контролируемый integration flow;
- есть клиентская бизнес-логика;
- нужен переиспользуемый общий модуль;
- задача требует точного технического контроля, который не выражается готовыми механизмами.

## Простое правило выбора

| Ситуация | Что проверяем первым | Когда писать custom code |
|---|---|---|
| Стандартная Jira/JSM настройка | Штатная конфигурация | Только если конфигурации недостаточно |
| Разовая административная задача | Встроенный скрипт ScriptRunner | Если встроенный механизм не покрывает сценарий |
| Типовой событийный сценарий | Встроенный listener ScriptRunner | Если нужен действительно уникальный listener |
| Простая внутренняя Jira-логика | HAPI | Если HAPI не покрывает задачу |
| Интеграция или уникальная бизнес-логика | Пользовательский скрипт | Если нужен свой сценарий и нет простого поддерживаемого пути |

## Вопросы перед созданием нового скрипта

Перед тем как создавать новый скрипт, ответьте на вопросы:

1. Это точно нельзя решить конфигурацией?
2. Нет ли встроенного скрипта ScriptRunner для этой задачи?
3. Нет ли встроенного listener ScriptRunner для этого событийного сценария?
4. Можно ли сделать это через HAPI без тяжелого кастомного слоя?
5. Это новый сценарий или просто изменение существующего?
6. Будет ли новый скрипт проще поддерживать, чем альтернативный путь?

Если хотя бы на первые четыре вопроса нет четкого ответа, новый custom script создавать рано.

## Анти-паттерны

### Анти-паттерн 1. Пишем скрипт, потому что "так привычнее"

Признаки:

- никто не проверял built-in возможности;
- никто не проверял HAPI;
- скрипт пишется быстрее, чем формулируется задача.

Почему плохо:

- плодится лишний код;
- растет стоимость поддержки;
- усложняется апгрейд и диагностика.

### Анти-паттерн 2. Разовая административная задача превращается в рабочий код

Признаки:

- разовая массовая операция оформляется как "настоящий сервис";
- код живет дольше самой задачи;
- никто не знает, зачем он остался.

Почему плохо:

- временный код случайно становится постоянным;
- навигация и ownership деградируют.

### Анти-паттерн 3. Новый скрипт вместо рефакторинга старого

Признаки:

- похожая логика уже есть;
- новый файл создается потому, что старый неудобный;
- вместо исправления структуры появляется дубль.

Почему плохо:

- знания размазываются по нескольким скриптам;
- одинаковые баги живут в нескольких местах;
- документация перестает совпадать с реальностью.

## Что писать на странице скрипта

Если новый custom script все же создается, на его странице должно быть явно видно:

- почему built-in или конфигурация не подошли;
- почему нужен именно custom path;
- можно ли потом упростить решение;
- нет ли у скрипта временного статуса.

Это особенно важно для:

- клиентских решений;
- listeners;
- mail handlers;
- migration scripts;
- operational scripts.

## Что добавляем в наш стандарт как правило

1. Новый кастомный скрипт не создается, пока не проверены более простые варианты.
2. Встроенные скрипты и встроенные listeners ScriptRunner рассматриваются как нормальный первый выбор.
3. HAPI рассматривается раньше тяжелого низкоуровневого API.
4. Разовая задача не должна автоматически становиться production-кодом.
5. Если достаточно доработки существующего скрипта, новый файл не создаем без причины.
