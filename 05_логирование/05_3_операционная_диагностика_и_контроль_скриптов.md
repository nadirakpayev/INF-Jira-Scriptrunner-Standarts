# 5.3 Операционная диагностика и контроль скриптов

> Эта страница описывает, как сопровождать ScriptRunner-скрипты после запуска в работу: где искать проблемы, как видеть деградацию и как контролировать изменения.

## Для чего нужна эта страница

Написать рабочий скрипт недостаточно. После запуска нужно понимать:

- где смотреть ошибки;
- как увидеть частые сбои;
- как заметить деградацию производительности;
- как проверить последствия изменений и апгрейдов;
- как понять, кто и что менял в конфигурации ScriptRunner.

Эта страница фиксирует минимальную операционную модель для рабочих скриптов.

## Основные источники диагностики

### 1. Рабочие логи

Логи нужны, чтобы понять:

- что запускалось;
- на каком шаге произошла проблема;
- какая сущность обрабатывалась;
- чем завершился сценарий.

Когда идти сюда:

- скрипт упал прямо сейчас;
- нужно понять причину конкретной ошибки;
- нужно увидеть частичные отклонения;
- нужно проверить, дошел ли сценарий до конца.

Когда логов недостаточно:

- если нужно понять деградацию во времени;
- если нужно понять, какие скрипты в принципе проблемные;
- если нужно увидеть устаревшие API и предупреждения статической проверки.

### 2. Execution History

`Execution History` нужен для анализа выполнения во времени.

Он полезен, если нужно:

- увидеть failure rate;
- сравнить длительность выполнения;
- понять, стал ли скрипт работать медленнее;
- увидеть корреляцию с релизом, апгрейдом или изменением данных.

Практический смысл:

- логи отвечают на вопрос `что случилось в конкретном запуске`;
- execution history отвечает на вопрос `что происходит со скриптом в динамике`.

### 3. Script Registry

`Script Registry` нужен для обзора всего ScriptRunner-контра:

- какие скрипты есть в системе;
- где они лежат;
- какие у них type-checking errors;
- какие есть предупреждения об устаревших API;
- что можно экспортировать для контроля состава инстанса.

Это особенно полезно:

- после апгрейда Jira;
- после апгрейда ScriptRunner;
- при ревизии legacy-кода;
- при подготовке к рефакторингу.

### 4. Audit Log

`Audit Log` нужен, чтобы видеть конфигурационные изменения ScriptRunner:

- кто добавил script;
- кто поменял script;
- кто удалил script;
- когда запускались встроенные скрипты ScriptRunner, которые могли изменить систему.

Это помогает разбирать вопросы уровня:

- кто изменил listener;
- почему поведение поменялось;
- было ли массовое изменение конфигурации.

### 5. Guardrails и встроенные операционные скрипты

Отдельный операционный контур дают встроенные скрипты ScriptRunner и guardrails:

- помогают видеть превышение рекомендуемых порогов;
- позволяют решать часть типовых админских задач без кастомного Groovy;
- полезны как preventive control, а не только как реакция на проблему.

## Что обязательно знать про каждый рабочий скрипт

Для рабочего скрипта должно быть понятно:

- где его код;
- где его страница документации;
- где смотреть его логи;
- есть ли execution history;
- кто владелец;
- как понять, что скрипт отработал успешно;
- как понять, что он деградировал;
- как проверить его после изменения.

Если этого нельзя понять быстро, операционная модель у скрипта слабая.

## Ежедневный операционный маршрут

Каждый день или при возникновении инцидента идем по такому маршруту:

1. Проверяем рабочие логи.
2. Смотрим, это разовый сбой или повторяющийся сценарий.
3. Если проблема повторяется, открываем execution history.
4. Проверяем, не было ли недавних изменений в конфигурации через audit log.
5. Если проблема началась после обновления или изменения API, проверяем Script Registry на предупреждения и использование устаревших API.

## Еженедельный контроль

Раз в неделю полезно проверять:

- какие jobs или listeners дают ошибки;
- у каких скриптов растет время выполнения;
- есть ли новые предупреждения или использование устаревших API;
- нет ли временных скриптов, которые уже должны быть выведены из active-конра;
- нет ли операционных сценариев, которые проще заменить встроенным скриптом ScriptRunner.

## Контроль после апгрейда

После апгрейда Jira или ScriptRunner минимальный порядок такой:

1. Проверить `Script Registry` на staging.
2. Посмотреть type-checking errors.
3. Посмотреть deprecation warnings.
4. Запустить критичные сценарии на тестовом контуре.
5. Проверить execution history у чувствительных jobs/listeners.
6. Отдельно проверить:
   - интеграции;
   - scheduled jobs;
   - mail handlers;
   - клиентские скрипты;
   - общие переиспользуемые модули.

## Какой сценарий считать проблемным

Скрипт считается операционно рискованным, если:

- он часто падает;
- failure rate вырос по сравнению с нормой;
- время выполнения растет;
- он зависит от внешней системы и это не видно в логах;
- после апгрейда появились предупреждения или устаревшие API;
- для него нет понятных шагов проверки;
- никто не может быстро объяснить, где искать причину сбоя.

## Если проблема уже случилась

Используйте такой порядок:

### Шаг 1. Понять тип проблемы

- полный сбой;
- частичная ошибка;
- деградация производительности;
- неожиданный побочный эффект;
- конфигурационное изменение.

### Шаг 2. Открыть правильный источник

- ошибка выполнения -> логи;
- повторяющиеся сбои -> execution history;
- предупреждения после обновления -> Script Registry;
- подозрение на изменение конфигурации -> audit log.

### Шаг 3. Зафиксировать результат

На странице скрипта должно появиться:

- краткое описание проблемы;
- дата;
- что нашли;
- как проверили;
- нужен ли отдельный follow-up.

## Что писать на странице скрипта заранее

Чтобы сопровождение было проще, на странице рабочего скрипта стоит держать блок:

`Операционные заметки`

Минимум в нем:

- где смотреть логи;
- какой нормальный результат выполнения;
- какой тревожный сигнал;
- есть ли execution history;
- какие внешние зависимости особенно критичны;
- что проверять после изменения или апгрейда.

## Чего не делаем

- Не считаем логи единственным источником правды.
- Не игнорируем deprecation warnings после апгрейда.
- Не оставляем скрипт без владельца и без понятного операционного маршрута.
- Не ждем боевого инцидента, чтобы впервые открыть execution history.
- Не отключаем audit log без сильной причины.

## Минимальный операционный чек-лист для критичного скрипта

- Есть понятные `INFO`, `WARN`, `ERROR` сообщения.
- Понятно, где смотреть execution history.
- Понятно, как найти скрипт через script registry.
- Понятно, как проверить последствия конфигурационных изменений.
- На странице скрипта есть шаги проверки после изменения.
- После апгрейда сценарий проходит повторную проверку.

## Что добавляем в наш стандарт как правило

1. У рабочего скрипта должен быть понятный операционный маршрут.
2. Для критичных скриптов нужно смотреть не только логи, но и execution history.
3. После апгрейда Script Registry проверяется обязательно на staging.
4. Конфигурационные изменения ScriptRunner должны быть наблюдаемы через audit log.
5. Встроенные скрипты и guardrails рассматриваются как часть операционного управления.
