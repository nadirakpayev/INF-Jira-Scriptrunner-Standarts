# 6.2 Правила интеграций с Assets и AQL

> Эта страница задает минимальные и практичные правила для интеграций, которые читают, ищут, создают или обновляют объекты в `Assets`.

## Для чего нужна эта страница

Интеграции с `Assets` быстро становятся сложными, потому что в них обычно смешиваются:

- внешний источник данных;
- схема объектов `Assets`;
- AQL-запросы;
- логика создания и обновления объектов;
- ссылки между объектами;
- частичные ошибки и повторные прогоны.

Если это не разложить заранее, скрипт начинает "как-то работать", но его уже трудно поддерживать и безопасно менять.

## Где эта страница применяется

Эти правила обязательны для сценариев уровня:

- `GLPI -> Jira Assets`;
- `CMDB Integration Playbook / Runbook`;
- клиентских синхронизаций, которые создают или обновляют объекты в `Assets`;
- скриптов, которые используют `AQL` для поиска связанных объектов, фильтрации или проверки ссылок.

## Главный принцип

В интеграциях с `Assets` сначала проектируем данные и запросы, и только потом пишем код.

Хорошая интеграция с `Assets` строится так:

1. Понимаем источник истины.
2. Понимаем, какие object types и attributes участвуют.
3. Определяем, как искать объект.
4. Определяем, когда создаем, а когда обновляем.
5. Определяем, что считаем частичной ошибкой.
6. Только после этого пишем скрипт.

## Что должно быть ясно до начала разработки

До начала разработки должно быть явно определено:

- какая внешняя система является источником истины;
- какие object types участвуют;
- какие attributes обязательны;
- какой атрибут или набор атрибутов является устойчивым идентификатором;
- какие reference types используются;
- что считается новым объектом;
- что считается обновлением существующего объекта;
- что делать, если объект найден неоднозначно;
- что делать, если часть данных пришла, а часть нет.

Если это не определено, писать код рано.

## Когда использовать AQL

`AQL` нужен, когда необходимо:

- найти объект по устойчивому признаку;
- ограничить выборку объектным типом;
- пройти по ссылкам между объектами;
- найти связанные объекты по reference type;
- выбрать объекты для сверки или массовой проверки;
- проверить связку `объект <-> Jira issue`.

`AQL` не должен превращаться в скрытый слой бизнес-логики, который невозможно быстро прочитать и проверить.

## Когда нужен REST API, а не обходные пути

Для интеграций с `Assets` используем поддерживаемые интерфейсы:

- `AQL` для поиска и фильтрации;
- официальный `Assets REST API` для чтения и изменения объектов;
- официальный `Jira REST API`, если нужно связывать интеграцию с issue-контуром Jira.

Не делаем так:

- не читаем `Assets` напрямую из базы данных;
- не строим интеграцию на неофициальных внутренних обходах, если есть поддерживаемый REST;
- не прячем схему поиска в случайных SQL-подобных хаках.

## Правила написания AQL

### 1. Ограничивайте запрос объектным типом

Если запрос можно сузить через `objectType`, `objectTypeId` или `objectTypeAndChildren()`, это нужно делать.

Плохо:

```text
"Serial Number" = "ABC-123"
```

Лучше:

```text
objectType = "Device" AND "Serial Number" = "ABC-123"
```

Такой запрос проще понимать и безопаснее поддерживать.

### 2. Используйте устойчивые технические атрибуты

Искать объект лучше по техническому идентификатору, внешнему ключу, серийному номеру или другому устойчивому признаку.

Не делайте ставку на:

- `label`, если это не реальный стабильный ключ;
- человекочитаемое имя, которое может меняться;
- свободный текст.

### 3. Помните, что AQL чувствителен к регистру в именах атрибутов

По официальной документации Atlassian:

- `AQL` чувствителен к регистру;
- имя атрибута должно существовать в схеме;
- значения и атрибуты с пробелами нужно брать в кавычки;
- кавычки внутри значения нужно экранировать.

Практический вывод для стандарта:

- не писать названия атрибутов "по памяти";
- не копировать запрос между схемами без проверки;
- хранить рабочий запрос на странице скрипта вместе с пояснением, что он делает.

### 4. Осторожно используйте dot notation

`Dot notation` удобен, когда нужно пройти по ссылке на связанный объект.

Пример класса задач:

```text
"Country"."Office" = "Stockholm"
```

Но чем глубже цепочка, тем хуже:

- тяжелее читать;
- тяжелее диагностировать ошибку;
- выше риск плохой производительности.

Для production-сценариев правило такое:

- `1-2` уровня нормально;
- глубокие цепочки нужно отдельно обосновывать;
- если в `order by` используется dot notation, это тоже считается нагрузкой.

### 5. Не используйте `anyAttribute` без необходимости

Официальная документация Atlassian отдельно предупреждает, что `anyAttribute` может замедлять поиск, особенно на больших установках.

Поэтому:

- для production-интеграций `anyAttribute` по умолчанию запрещен;
- допустим только как временный диагностический прием;
- если он все-таки нужен, это должно быть явно объяснено на странице скрипта.

### 6. Используйте reference functions осознанно

Функции `inR()` и `outR()` полезны, когда нужно искать объекты через связи.

Хороший подход:

- явно ограничивать `objectType`;
- по возможности указывать `refType`;
- не писать "магические" reference-запросы без объяснения.

Пример рабочего паттерна:

```text
object having inR(objectType = "File System", refType IN ("Installed"))
```

Это читается лучше, чем неограниченный запрос "по всем входящим ссылкам".

### 7. Используйте `order by` только когда это действительно нужно

Сортировка нужна не всегда.

Если порядок не влияет на бизнес-результат, лучше не усложнять запрос.

Если сортировка нужна:

- фиксируйте, зачем именно она нужна;
- не стройте тяжелую сортировку через длинную цепочку ссылок без необходимости.

## Базовые паттерны AQL, которые допустимо использовать

### Поиск по объектному типу и ключевому атрибуту

```text
objectType = "Device" AND "Serial Number" = "ABC-123"
```

Подходит для:

- проверки существования объекта;
- поиска кандидата на update;
- идемпотентного create-or-update сценария.

### Поиск по ссылке на родительский объект

```text
"Host"."Name" = "srv-app-01"
```

Подходит для:

- работы с зависимыми сущностями;
- поиска объекта через известный родительский контур.

### Поиск по входящим или исходящим связям

```text
object having outR(objectType = "Employees", refType IN ("Location", "Country"))
```

Подходит для:

- проверки reference-модели;
- выборки связанных объектов по понятному refType.

### Поиск связанных Jira tickets

```text
object having connectedTickets("project = ITSM AND statusCategory != Done")
```

Подходит для:

- operational checks;
- отчетных и контрольных сценариев;
- поиска объектов, которые реально участвуют в процессе.

## Архитектурный шаблон интеграции с Assets

Для сценариев уровня `GLPI -> Jira Assets` или `CMDB Integration Playbook / Runbook` рекомендуемый поток такой:

1. Получить данные из внешней системы.
2. Провести нормализацию и валидацию.
3. Определить object type и обязательные attributes.
4. Выполнить AQL-поиск кандидата на update.
5. Сравнить текущее состояние и новое состояние.
6. Выполнить `create`, `update`, `skip` или `flag for review`.
7. Зафиксировать итог по количеству обработанных сущностей, конфликтов и ошибок.

Такой поток лучше, чем "в одном цикле искать, создавать, обновлять и логировать без явных стадий".

## Что обязательно документировать на странице скрипта

Для каждого `Assets`-скрипта на странице в Confluence должно быть явно указано:

- какие object types используются;
- какие attributes являются ключевыми;
- какие `AQL`-запросы используются и для чего;
- что считается критерием поиска существующего объекта;
- когда выполняется `create`;
- когда выполняется `update`;
- какие reference types создаются или читаются;
- что считается частичной ошибкой;
- что логируется как итог выполнения;
- как проверить корректность результата после изменения.

Если в скрипте больше одного AQL-запроса, их нужно хранить как именованные запросы, а не как россыпь строк по всему коду.

## Минимум без бюрократии

Для обычной рабочей интеграции достаточно этого минимума:

1. Понятно, какие object types участвуют.
2. Понятно, по какому атрибуту ищется объект.
3. AQL-запрос читаем и ограничен по области поиска.
4. Понятно, где `create`, где `update`, где `skip`.
5. В логах есть итог по обработанным объектам и ошибкам.
6. На странице скрипта есть рабочие примеры запросов и шаги проверки.

## Что не делаем

- Не ищем production-объекты через `anyAttribute`, если можно использовать точный атрибут.
- Не используем `label` как технический ключ без явного обоснования.
- Не пишем длинный AQL прямо внутри бизнес-логики без имени и пояснения.
- Не смешиваем поиск, маппинг, create/update и reference-логику в один непрозрачный блок.
- Не работаем с `Assets` через прямой доступ к БД, если задачу решает поддерживаемый API.
- Не считаем "ничего не нашли" автоматическим поводом создать новый объект без проверки на неоднозначность.
- Не держим сложный reference-query без объяснения reference type.

## Короткий чек-лист перед публикацией

- Запрос ограничен по `objectType` или другой ясной области поиска.
- Запрос использует реальные атрибуты схемы.
- Поиск опирается на устойчивый идентификатор.
- Понятно, что происходит при `0`, `1` и `>1` найденных объектах.
- Логирование показывает итог по объектам, а не только старт процесса.
- Страница скрипта содержит сами запросы и объяснение их роли.
