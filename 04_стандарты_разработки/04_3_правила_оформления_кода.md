# 4.3 Правила оформления кода

> Код должен быть не только рабочим, но и понятным.

## Именование

- имя должно отражать действие или смысл;
- односимвольные имена допустимы только в очень локальном контексте;
- технические сокращения без расшифровки запрещены;
- если сущность связана с внешней системой, это должно быть видно в названии.

Хорошо:

```groovy
def glpiDevices = loadGlpiDevices()
def matchedAsset = findAssetByExternalId(deviceId)
```

Плохо:

```groovy
def d = load()
def x = find(deviceId)
```

## Комментарии

- комментарий нужен только там, где смысл неочевиден;
- комментарий не должен пересказывать код;
- временные комментарии и TODO без владельца не допускаются.

Хорошо:

```groovy
// Не обновляем объект, если источник не прислал обязательный внешний идентификатор.
if (!externalId) {
    return
}
```

Плохо:

```groovy
// Проверяем, что externalId не пустой.
if (!externalId) {
    return
}
```

## Функции

- функция должна делать одну понятную вещь;
- слишком длинные функции надо дробить;
- скрытые глобальные зависимости нежелательны;
- побочный эффект должен быть очевиден.

Хорошо:

```groovy
def mapGlpiDevice(sourceDevice) {
    return [
        externalId: sourceDevice.id,
        name      : sourceDevice.name
    ]
}
```

Плохо:

```groovy
def handleEverything(device) {
    // чтение, маппинг, валидация, запись, логирование и обработка ошибок
}
```

## Ошибки

- ошибки нельзя глотать молча;
- если ошибка ожидаемая, это должно быть видно;
- сообщение об ошибке должно помогать поддержке.

Хорошо:

```groovy
try {
    updateAsset(assetPayload)
} catch (Exception ex) {
    log.error("GLPI sync failed on asset update: externalId=${assetPayload.externalId}", ex)
    throw ex
}
```

Плохо:

```groovy
try {
    updateAsset(assetPayload)
} catch (Exception ignored) {
}
```

## Читаемость

- длинные цепочки логики нужно раскладывать на понятные шаги;
- вложенность должна быть умеренной;
- повторяемый код нужно сокращать через общие функции, но без лишней абстракции.

Хорошо:

- сначала фильтрация;
- потом основная обработка;
- потом запись результата;
- потом финальный лог.

Плохо:

- три уровня вложенности;
- переменные с короткими неочевидными именами;
- важные побочные эффекты разбросаны по файлу.
